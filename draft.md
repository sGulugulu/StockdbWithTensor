# 基于张量分解的股票因子降维与模式发现 Draft

## 1. 文档目的

这份 `draft.md` 用来统一当前毕设项目的研究目标、实验主线、系统边界、历史复盘和未来路线。它的作用是作为后续 `humanize-gen-plan`、分支实施计划和 RLCR 执行的上游草稿，而不是直接展开到实现细节。

它需要解决的核心问题是：

1. 这个项目本质上要证明什么。
2. 当前论文与系统之间应该如何统一叙事。
3. 当前正式实验范围与长期股票覆盖范围如何区分。
4. 当前最重要的任务树应该如何拆分。

## 2. 项目定位

这个项目既不是单纯的选股系统，也不是完全脱离实现的纯论文工作。更准确的定位应该是：

1. 论文主线聚焦于“基于张量分解的股票因子降维与模式发现”。
2. 实验系统负责把方法跑通、验证并沉淀结果。
3. Web 系统负责把实验能力、结果合同和查询能力变成可展示、可调用、可复现的系统接口。

因此，论文、实验系统、Web 系统并不是三条互相独立的线，而是同一个研究问题的三个表达层：

1. 论文表达学术问题与方法。
2. 实验系统表达方法的可运行性和可验证性。
3. Web 系统表达结果的可查询性和可展示性。

## 3. 研究核心对象与方法

### 3.1 统一研究对象

整个项目的统一研究对象是：

- `股票-因子-时间` 三维张量

即：

1. 第一维：股票
2. 第二维：因子
3. 第三维：时间

这个三维张量是论文方法、实验设计、结果解释和系统接口的共同核心对象。

### 3.2 主要方法路径

当前论文与实验的核心方法固定为：

1. `CP` 分解
2. `Tucker` 分解

这两种方法的定位是：

1. 首先用于因子降维。
2. 其次用于模式发现。
3. 最后才在预测或决策阶段接受有效性检验。

也就是说，项目主问题不是“做一个收益预测模型”，而是研究：

- 如何基于三维张量完成股票因子降维；
- 如何从分解结果中发现股票、因子、时间之间的潜在模式；
- 这些模式在后续窗口中是否仍有解释或应用价值。

### 3.3 参考依据

方法方向主要参考 `./参考文献` 下的 PDF 文献，相关文献覆盖：

1. 张量方法综述
2. 张量分解理论
3. 金融时间序列中的张量方法
4. 股票应用相关研究

这些文献构成论文方法章节和实验设计章节的重要理论依据。

## 4. 范围边界

### 4.1 当前正式实验样本

当前正式实验主要围绕以下三个指数开展：

1. `HS300`
2. `SZ50`
3. `ZZ500`

它们是当前论文和系统主实验的正式样本，不等于系统长期只支持这三个股票池。

### 4.2 长期股票覆盖范围

系统长期股票覆盖范围仍然保留更广的 A 股范围。也就是说，架构上必须明确区分：

1. 长期股票覆盖范围
2. 当前实验样本层
3. 当前重点指数实验配置

因此：

- 当前正式实验以三个指数为主；
- 但系统和数据底座不能把长期股票范围写死成这三个指数。

### 4.3 正式时间窗口

当前 formal 数据与论文主实验所依赖的正式时间窗口固定为：

- `2015-01-01` 到 `2026-04-01`

这个时间窗口需要在文档、配置和实验中保持一致，不能继续漂移成“当前最新可用日期”。

## 5. 统一叙事框架

总体路线图采用“双主线叙事”，但实际优先级上系统实现仍先于论文交付：

### 5.1 实验样本验证

当前正式实验在代表性指数样本上开展，目标是验证：

1. 张量分解是否能够有效完成因子降维。
2. 是否能够从结果中发现可解释的模式。
3. 这些模式在后续窗口中是否仍然有价值。

### 5.2 未来决策应用

系统长期保留更广股票范围，意味着方法一旦验证成立，应当能够迁移到更广股票范围的决策支持场景。

因此，论文不应写成普通股票系统说明书，而应写成：

1. 以 `股票-因子-时间` 三维张量为统一研究对象。
2. 以 `CP` / `Tucker` 为核心方法。
3. 以当前正式实验样本完成方法验证。
4. 以未来更广应用作为受控扩展方向。

## 6. 实验样本协议

实验层必须区分长期股票范围与当前实验配置。

### 6.1 长期覆盖范围

更广的 A 股范围仍然保留在系统和数据底座层，不因为当前论文实验样本较少而被删除。

### 6.2 当前主要实验样本

当前主要实验围绕：

1. `HS300`
2. `SZ50`
3. `ZZ500`

这三个指数分别建实验，不先混合成单一总样本池。

### 6.3 切分策略

系统层必须支持可配置切分策略：

1. 比例可配置
2. 维度可配置
3. 支持按时间切分、按股票切分、混合切分

### 6.4 论文默认切分

论文主实验默认采用**按时间切分**。其他切分方式保留为系统能力和后续稳健性实验方向，不强行塞进当前论文主结果。

## 7. 数据预处理与信息泄露控制

数据预处理必须被提升为原始 formal 数据与张量输入之间的独立阶段。它至少包括：

1. 样本筛选
2. 时间与字段对齐
3. 缺失值处理
4. 异常值处理
5. 因子方向统一与标准化
6. 标签与元信息拆分

这里的硬约束是：

1. 任何未来信息不得穿越训练 / 预测边界。
2. 未来收益标签只作为评价目标，不进入输入张量。

## 8. 实验主线

实验主线分为三层：

### 8.1 张量构造

按当前实验样本分别构造 `股票-因子-时间` 三维张量。

### 8.2 张量分解

应用 `CP` 和 `Tucker`。

### 8.3 验证

比较：

1. 分解质量
2. 模式发现能力
3. 在后续窗口中的预测或决策有效性

论文中必须避免把 `CP` / `Tucker` 降格成普通收益预测工具。它们首先是因子降维与模式发现方法。

## 9. 评估框架

评价体系建议分三层：

### 9.1 分解质量

例如：

1. 重构质量
2. rank 行为
3. 稳定性

### 9.2 模式发现与解释

例如：

1. 因子贡献
2. 股票结构
3. 时间模式
4. 不同方法的解释差异

### 9.3 预测或决策有效性

例如：

1. 训练阶段提取出的结构在后续窗口中是否仍有价值
2. 不同样本池上的泛化表现
3. 候选股票排序效果

这三层同时服务：

1. 论文结果章节
2. 系统结果页与 API 合同

## 10. 系统边界

长期目标架构固定为：

1. 纯 Go API 网关
2. Python 作为实验执行器和数据处理运行时
3. DuckDB 作为 formal 数据查询层和 run 归档层
4. `code/outputs` 作为 run 结果产物层

### 10.1 Go 职责

1. 提供 HTTP 服务
2. 做请求校验
3. 提供统一响应体
4. 管理任务生命周期
5. 持久化状态
6. 做归档
7. 聚合 DuckDB 查询与结果文件读取

### 10.2 Python 职责

1. 解析 run 请求
2. 生成 `submitted_config.yaml`
3. 执行实验
4. 将结果写入 `code/outputs/<run_id>/`

## 11. API 与运行设计

### 11.1 统一响应包裹结构

Go 网关统一返回如下包裹结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "req_xxx",
  "timestamp": "2026-04-09T00:00:00Z"
}
```

### 11.2 路由分组

建议的路由分组：

1. run 提交与生命周期管理
2. run 结果查询
3. formal 数据查询

### 11.3 默认提交模型

`POST /api/runs` 默认异步：

1. 校验请求
2. 分配 `run_id`
3. 写入运行态
4. 返回 `202 Accepted`
5. 异步启动 Python runner
6. 由前端轮询状态或详情接口

### 11.4 实验配置模型

一个 run 请求至少应当包含：

1. config profile
2. 训练样本配置
3. 预测样本配置
4. 切分策略
5. 模型设置
6. 输出设置

因此系统设计应先定义实验配置模型，再定义路由，而不是反过来。

### 11.5 状态真源

Go 持有运行状态真源，采用双层结构：

1. 运行态 JSON，例如 `var/runs/<run_id>.json`
2. DuckDB 归档表，例如 `gateway_runs`、`gateway_run_events`

Python 不反向覆盖 Go 的状态真源。

## 19. 依赖关系

1. **研究数据与实验底座** 是上游基础
